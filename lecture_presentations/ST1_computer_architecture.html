>> The hook

A computer can only do one thing.

It moves numbers around and does simple arithmetic on them — add, compare, copy. That's it. No "open browser", no "play video", no "run Python". Just tiny operations on numbers, done billions of times a second.

Everything else — your game, your music, your Python program — is that one trick, stacked into towers so tall they look like magic. Today we learn the trick.

Keep asking: "where do the numbers live, and what's doing the arithmetic?" Answer that and you understand any computer.
>> 01 / 06 · The big picture

Every computer is these four parts

Your laptop, your phone, the server running this slide — same blueprint, drawn in 1945 and still true.

CPU
the worker — does the math
↔
Memory (RAM)
the desk — holds what's in use now
Storage
the drawer — keeps things long-term
↔
Input / Output
keyboard, screen, network — the outside world
The bus is the set of wires connecting them — the hallway data travels down. When people say a computer is "fast", they usually mean some mix of these four and the bus between them.
>> 02 / 06 · Everything is bits

One wire, two states — that's the whole alphabet

it_is_all_numbers
# A bit = one wire: ON (1) or OFF (0)
# 8 bits = 1 byte = one of 256 values

01001000  → the number 72  → the letter 'H'
01101001  → the number 105 → the letter 'i'

# Same bits, different meaning:
#   as text   → "Hi"
#   as a pixel→ a shade of grey
#   as sound  → a speaker position
It's all the same stuff.
Text, images, video, your Python file — every one is just bytes. The meaning comes from the program reading them.

>> 03 / 06 · The CPU

Fetch · Decode · Execute — forever

The CPU runs one tiny loop, over and over. Each lap handles one instruction.

FETCH
grab next instruction from memory
→
DECODE
figure out what it means
→
EXECUTE
do it (add, copy, compare…)
↳ then back to FETCH for the next one → repeat ~3,000,000,000 times per second
"3.0 GHz" means 3 billion of these laps every second. A "core" is one worker running this loop; a 4-core CPU runs four of these loops at once.
>> The CPU, a little closer

The clock, the cores, and the scratchpad

Clock speed

How many laps per second. 3 GHz = 3 billion/sec. Higher = each core does more work per second.

Cores

How many laps at once. 8 cores = 8 workers running the fetch-decode-execute loop in parallel.

Registers

A handful of tiny slots inside the CPU holding the numbers it's working on right this instant. The fastest memory there is.

Why not just buy 100 GHz? Physics. Faster clocks run hotter and hit limits, so since ~2005 makers add more cores instead of more GHz. That's why "use all your cores" became a thing programmers care about.
>> 04 / 06 · The memory hierarchy

Fast costs money, so you get a little of it

The CPU needs data instantly, but fast memory is tiny and expensive. So computers stack layers: small & blazing at the top, huge & slow at the bottom.

Registers inside the CPU ~1 ns · a few bytes
Cache (L1–L3) on the CPU chip ~5 ns · a few MB
RAM the sticks on the board ~100 ns · 8–32 GB
SSD / Disk long-term storage ~100,000 ns · 256 GB–2 TB
Reading from disk vs a register is like a 3-month wait vs a 1-second one. That gap is why programs try hard to keep hot data near the top.
>> 05 / 06 · The classic confusion

RAM vs Storage — not the same thing

RAM — the desk
  • Working space — holds what's open right now.
  • Fast, but forgets everything when power is cut.
  • Measured in GB: 8, 16, 32.
  • "My laptop is slow with 20 tabs" → out of RAM.
STORAGE — the drawer
  • Long-term shelf — your files, photos, OS.
  • Slower, but remembers with the power off.
  • Measured in GB/TB: 256 GB, 1 TB.
  • "I'm out of space for photos" → out of storage.
The bridge to your code: opening a file reads it from storage into RAM. Saving writes it from RAM back to storage. That's literally what open(...) and .write() do.
>> 06 / 06 · Bringing it home

Your Python, all the way down

The CPU can't read print("hi"). So Python translates, step by step, until it's numbers the CPU understands.

from_you_to_the_chip
"print('hi')"          # 1. You write this — text a human reads
      ↓
bytecode                # 2. Python compiles to simple steps (LOAD, CALL...)
      ↓
interpreter             # 3. The Python program runs each step...
      ↓
machine code            # 4. ...which the CPU runs as raw instructions
      ↓
01011010 11000001     # 5. fetch → decode → execute → "hi" on screen
WHY YOU CARE This is why Python is "slower" than C — those extra translation layers buy you readability. Usually a great trade.
>> Try it yourself

Two-minute explorations

LOOK
Find your own specs

Open your system info. Find your CPU cores & GHz, your RAM, and your storage. Label each with which of today's four parts it is.

THINK
Explain the slowdown

Stretch. In Python, run import os; os.cpu_count(). That number is how many fetch-decode-execute loops your machine can run at once.
